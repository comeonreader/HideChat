# 类微信实时聊天渐进式演进方案 v1.0

## 1. 改动目的

在 **不改核心业务流程、尽量不改 schema、优先复用现有模块** 的前提下，为 HideChat 设计并分阶段落地一套更稳定的类微信实时聊天架构，重点提升：

1. WebSocket 连接稳定性
2. 消息发送与 ACK 一致性
3. 前端聊天状态可维护性
4. 断线重连后的数据自愈能力

本方案默认采用 **渐进演进**，即：

* 保留现有 HTTP 发消息 / 拉历史 / 标记已读能力
* 保留现有 Spring WebSocket 与前端 React 页面组织
* 先整理连接与协议层，再逐步补齐消息状态、去重与增量同步

---

## 2. 现状问题

结合当前代码，实时链路已经可用，但还存在几个稳定性问题：

### 2.1 前端连接管理分散

当前 [frontend/src/hooks/useChatRealtime.ts](/home/reader/HideChat/frontend/src/hooks/useChatRealtime.ts) 直接持有 `WebSocket` 实例，并在 Hook 内同时处理：

* 连接建立
* 收包解析
* 已读同步
* 失败回退 HTTP

这会导致：

* 连接生命周期与页面生命周期耦合过深
* 页面重渲染时不易保证“单例连接”
* 重连、心跳、前后台感知难以持续扩展

### 2.2 协议类型缺少统一 envelope

当前前后端使用 `CHAT_SEND / CHAT_ACK / CHAT_RECEIVE / CHAT_READ / CHAT_ERROR` 这类散落字符串，后端 [backend/src/main/java/com/hidechat/websocket/handler/ChatWebSocketHandler.java](/home/reader/HideChat/backend/src/main/java/com/hidechat/websocket/handler/ChatWebSocketHandler.java) 直接按字符串分支处理，前端 `App.tsx` 和 `useChatRealtime.ts` 也直接按 type 写条件判断。

这会导致：

* 协议演进成本高
* 新增类型时容易漏改
* 不利于统一日志、错误码、traceId、ackId

### 2.3 发送态与 ACK 对齐不完整

当前前端 `ChatMessage` 只有 `serverStatus`，本地发送时复用 `messageId` 作为 optimistic key。后端 `SendMessageRequest` 也直接允许传入 `messageId` 并落库。

问题在于：

* `messageId` 同时承担“客户端临时主键”和“服务端权威主键”语义，不够清晰
* ACK 丢失后，前端难以可靠判断消息是否已被服务端接收
* 重连重发时，去重边界不清晰

### 2.4 断线后只有“重新拉历史”的弱自愈

当前架构更多依赖：

* 在线时实时推送
* 离线后靠历史接口补偿

但缺少明确的 **增量同步协议**，因此：

* 重连后只能全量或手工刷新
* 前端无法清楚知道“从哪个时点开始补”
* 未知会话、漏 ACK、乱序推送都难以统一收敛

---

## 3. 演进目标

本次演进按四个阶段推进：

### 第一阶段：稳定连接层

实现单例 WebSocket 连接管理器、连接状态机、心跳 `ping/pong`、指数退避重连、前后台与 `online/offline` 感知。

### 第二阶段：统一实时协议

引入统一 envelope，支持以下消息类型：

* `auth`
* `auth_ok`
* `ping`
* `pong`
* `message_send`
* `message_ack`
* `error`

兼容现有链路，逐步替换 `CHAT_*` 字符串协议。

### 第三阶段：补齐消息一致性

引入 `clientMessageId`、消息状态 `sending/sent/failed`、ACK 对应更新、消息去重。

### 第四阶段：重连增量同步

引入 `sync_request / sync_response`，支持重连后增量追平，避免靠整页刷新自愈。

## 4. 总体架构

### 4.1 架构分层

```text
UI 页面 / 组件
    ↓
聊天应用编排层（App.tsx / useChatState）
    ↓
实时编排层（useChatRealtime）
    ↓
实时连接层（单例 ConnectionManager + 状态机 + 重连策略）
    ↓
协议层（Envelope 编解码、类型守卫、ACK 路由）
    ↓
消息状态层（pendingMap、去重索引、sync cursor）
    ↓
HTTP 补偿层（history / read / conversation/list）
```

后端对应分层：

```text
WebSocketConfig
    ↓
ChatWebSocketHandler
    ↓
Envelope 解析 / 类型分发
    ↓
连接会话管理 / 心跳 / 鉴权
    ↓
MessageService / ConversationService
    ↓
DB + Redis
```

### 4.2 核心原则

1. **连接与业务解耦**
   `WebSocket` 生命周期不再由页面临时状态直接驱动，而由单例连接管理器统一驱动。

2. **协议先统一，再扩展行为**
   先把 envelope 稳定下来，再逐步增加 ACK、去重、同步能力。

3. **服务端权威，前端可回放**
   服务端继续负责权威消息 ID 与最终状态；前端保留 optimistic UI，但必须能被 ACK 或 sync 校正。

4. **实时失败可回退**
   实时链路失败时，现有 HTTP 能继续兜底，不破坏主链路。

## 6. 连接状态机设计

### 6.1 状态定义

前端建议定义如下状态：

* `idle`
* `connecting`
* `authenticating`
* `connected`
* `reconnecting`
* `offline`
* `closing`
* `closed`
* `error`

### 6.2 状态流转

```text
idle
  → connecting
  → authenticating
  → connected

connected
  → reconnecting      （socket close / pong timeout）
  → offline           （浏览器 offline）
  → closing           （主动退出）

offline
  → reconnecting      （online 事件恢复）

reconnecting
  → connecting
  → authenticating
  → connected
  → error             （达到最大重试阈值）

closing
  → closed
```

### 6.3 状态机规则

1. 同一时刻只允许一个活动连接实例。
2. 页面切换不应重复创建 socket。
3. `visibilitychange` 到后台时，不主动断连，但降低心跳频率。
4. 浏览器 `offline` 时，立即进入 `offline`，暂停重连计时。
5. 浏览器 `online` 时，立即触发一次重连尝试。

---

## 7. 心跳与重连策略

### 7.1 心跳

建议采用应用层心跳而不是仅依赖底层 TCP：

* 客户端每 `25s` 发送一次 `ping`
* 服务端收到后立即返回 `pong`
* 连续 `2` 个心跳周期未收到 `pong`，判定连接失活

后台标签页可放宽为：

* `pingInterval = 55s`
* `pongTimeout = 20s`

### 7.2 指数退避

建议退避策略：

* 第 1 次：`1s`
* 第 2 次：`2s`
* 第 3 次：`4s`
* 第 4 次：`8s`
* 第 5 次及以后：`15s`

加入 `0~20%` 随机抖动，避免同批用户同时重连。

### 7.3 重连触发源

以下情况触发重连：

* socket `close`
* socket `error`
* `pong` 超时
* `online` 事件恢复
* 页面从后台切回前台且连接已失活

以下情况不自动重连：

* 用户登出
* token 明确失效且 refresh 失败
* 服务端返回不可恢复鉴权错误

## 8. 协议定义

## 8.1 通用 Envelope

```json
{
  "type": "message_send",
  "requestId": "req_123",
  "timestamp": 1712620800000,
  "traceId": "trace_abc",
  "data": {}
}
```

字段说明：

* `type`：消息类型
* `requestId`：本次请求唯一标识，便于日志与 ACK 对齐
* `timestamp`：发送时间
* `traceId`：链路追踪 ID，可选
* `data`：具体负载

## 8.2 `auth`

客户端连接建立后立即发送：

```json
{
  "type": "auth",
  "requestId": "req_auth_1",
  "timestamp": 1712620800000,
  "data": {
    "accessToken": "jwt-token"
  }
}
```

说明：

* 阶段一可继续兼容 query token
* 阶段二开始以 envelope `auth` 为主，query token 仅做兼容过渡

## 8.3 `auth_ok`

```json
{
  "type": "auth_ok",
  "requestId": "req_auth_1",
  "timestamp": 1712620800010,
  "data": {
    "userUid": "u_1001",
    "serverTime": 1712620800010,
    "heartbeatIntervalMs": 25000,
    "sessionId": "ws_s_1001"
  }
}
```

## 8.4 `ping`

```json
{
  "type": "ping",
  "requestId": "req_ping_1",
  "timestamp": 1712620825000,
  "data": {
    "clientTime": 1712620825000
  }
}
```

## 8.5 `pong`

```json
{
  "type": "pong",
  "requestId": "req_ping_1",
  "timestamp": 1712620825010,
  "data": {
    "serverTime": 1712620825010
  }
}
```

## 8.6 `message_send`

```json
{
  "type": "message_send",
  "requestId": "req_msg_1",
  "timestamp": 1712620830000,
  "data": {
    "clientMessageId": "cm_1001",
    "conversationId": "c_1001",
    "receiverUid": "u_1002",
    "messageType": "text",
    "payloadType": "plain",
    "payload": "hello",
    "fileId": null,
    "clientMsgTime": 1712620830000
  }
}
```

## 8.7 `message_ack`

```json
{
  "type": "message_ack",
  "requestId": "req_msg_1",
  "timestamp": 1712620830010,
  "data": {
    "clientMessageId": "cm_1001",
    "messageId": "m_9001",
    "conversationId": "c_1001",
    "status": "sent",
    "serverMsgTime": 1712620830010
  }
}
```

说明：

* `clientMessageId` 是前端更新 optimistic message 的主键
* `messageId` 是服务端权威消息 ID
* 阶段三起前端只以 `clientMessageId` 更新本地发送态

## 8.8 `error`

```json
{
  "type": "error",
  "requestId": "req_msg_1",
  "timestamp": 1712620830011,
  "data": {
    "code": 420201,
    "message": "接收方不属于当前会话",
    "retryable": false
  }
}
```

## 8.9 `sync_request`

```json
{
  "type": "sync_request",
  "requestId": "req_sync_1",
  "timestamp": 1712620900000,
  "data": {
    "sinceCursor": "1712620800010",
    "conversationIds": ["c_1001", "c_1002"]
  }
}
```

## 8.10 `sync_response`

```json
{
  "type": "sync_response",
  "requestId": "req_sync_1",
  "timestamp": 1712620900100,
  "data": {
    "nextCursor": "1712620900100",
    "messages": [],
    "hasMore": false
  }
}
```

## 9. 数据结构设计

## 9.1 前端 ChatMessage

建议从当前结构演进为：

```ts
interface ChatMessage {
  messageId: string;
  clientMessageId?: string | null;
  conversationId: string;
  senderUid: string;
  receiverUid: string;
  payload: string;
  messageType: string;
  payloadType?: string;
  fileId?: string | null;
  clientMsgTime?: number | null;
  serverMsgTime: number;
  serverStatus?: string;
  deliveryStatus?: "sending" | "sent" | "failed";
}
```

规则：

* `messageId`：服务端权威 ID
* `clientMessageId`：客户端发送幂等键
* `serverStatus`：服务端业务状态，如 `delivered / read`
* `deliveryStatus`：前端发送生命周期状态，如 `sending / sent / failed`

## 9.2 前端实时状态

```ts
interface RealtimeState {
  connectionState: "idle" | "connecting" | "authenticating" | "connected" | "reconnecting" | "offline" | "closing" | "closed" | "error";
  retryCount: number;
  isOnline: boolean;
  isForeground: boolean;
  lastConnectedAt?: number | null;
  lastPingAt?: number | null;
  lastPongAt?: number | null;
  syncCursor?: string | null;
}
```

## 9.3 后端消息发送 DTO

建议在不改核心业务模型的前提下，把现有 `SendMessageRequest` 补充为：

```java
public class SendMessageRequest {
    private String clientMessageId;
    private String conversationId;
    private String receiverUid;
    private String messageType;
    private String payloadType;
    private String payload;
    private String fileId;
    private Long clientMsgTime;
}
```

说明：

* 现有 `messageId` 入参逐步废弃
* 服务端统一生成 `messageId`
* ACK 回包显式带回 `clientMessageId`

## 9.4 Redis 幂等键

建议 TTL 取 `10~30min`：

```text
ws:msg:dedupe:{senderUid}:{clientMessageId} -> serverMessageId
```

作用：

* 避免断线重发产生重复落库
* 不改数据库 schema 也能先建立幂等窗口

## 10. 分阶段落地顺序

## 10.1 第一阶段：连接管理器

### 目标

先解决“连得稳”问题，不动消息业务语义。

### 前端落点

1. `api/realtime.ts` 抽出单例连接管理器
2. `useChatRealtime.ts` 改为订阅式编排
3. `types/index.ts` 增加连接状态定义
4. `App.tsx` 只消费状态，不再直接依赖 socket 实例

### 后端落点

1. `ChatWebSocketHandler` 支持 `ping/pong`
2. `WebSocketSessionRegistry` 补充 session 元信息
3. 服务端保留现有 query token 鉴权

### 兼容策略

* 保持 `CHAT_SEND / CHAT_READ` 不变
* 先补 `ping / pong`
* 先不改消息 DTO

## 10.2 第二阶段：统一 Envelope

### 目标

建立统一协议壳，减少散落字符串。

### 实施方式

1. 前端新增 `RealtimeEnvelope`
2. 后端新增 `RealtimeEnvelopeDTO`
3. `ChatWebSocketHandler` 改为统一解包后分发
4. 对旧 `CHAT_*` 协议保留一个兼容映射层

### 兼容映射建议

* `CHAT_SEND` -> `message_send`
* `CHAT_ACK` -> `message_ack`
* `CHAT_ERROR` -> `error`

## 10.3 第三阶段：ACK 与去重

### 目标

让发送中的消息有稳定生命周期。

### 实施方式

1. 前端生成 `clientMessageId`
2. 本地消息初始为 `deliveryStatus = sending`
3. 服务端入库成功后返回 `message_ack`
4. 前端按 `clientMessageId` 把消息改为 `sent`
5. 超时未 ACK 的消息标为 `failed`
6. 手动重发时复用原 `clientMessageId`
7. Redis 做短期幂等窗口

### 去重规则

前端：

* 同一 `messageId` 只保留一条
* 同一 `clientMessageId` 的本地 pending 与 ACK 后的服务端消息要合并

后端：

* 同一用户、同一 `clientMessageId` 在 TTL 窗口内只处理一次

## 10.4 第四阶段：增量同步

### 目标

解决重连后丢消息、漏 ACK、未知会话问题。

### 实施方式

1. 前端保存 `syncCursor`
2. 每次 `auth_ok` 或重连成功后自动发 `sync_request`
3. 服务端按 `sinceCursor` 返回增量消息
4. 前端执行：
   * 去重
   * 合并消息状态
   * 更新会话列表
   * 刷新 `syncCursor`

### 回退策略

若 `sync_response` 异常：

* 回退到现有 `conversation/list + history` 拉取逻辑
* 不阻塞聊天主链路

## 12. 是否涉及接口 / 数据库 / 本地存储

### 12.1 是否涉及接口变更

涉及，但建议 **分阶段兼容**：

* WebSocket 协议会新增 envelope 与新 type
* HTTP 主链路可保持不变
* WebSocket 旧 `CHAT_*` type 在过渡期继续兼容

### 12.2 是否涉及数据库变更

**默认第一到第三阶段不改数据库 schema。**

理由：

* `clientMessageId` 可先通过 Redis 幂等键承接
* `syncCursor` 可先基于现有 `server_msg_time` 与消息历史查询能力实现

只有在后续需要“永久幂等审计”时，再评估新增字段或索引。

### 12.3 是否涉及前端本地存储结构变更

涉及，建议在 IndexedDB 缓存结构中补充可选字段：

* `clientMessageId`
* `deliveryStatus`
* `lastSyncCursor`

这属于 **向后兼容的本地结构扩展**，不需要破坏现有缓存。

## 13. 风险点

### 13.1 单例连接与 React 生命周期冲突

如果 manager 仍被 Hook 私有化，容易出现多实例连接。必须把 manager 提升到模块级单例或 store 持有。

### 13.2 ACK 与历史消息双写合并

一条消息可能先收到 ACK，后收到 sync；也可能先 sync，后 ACK。前端必须按 `clientMessageId + messageId` 合并，不能简单 append。

### 13.3 后台标签页定时器不准

浏览器后台页会降频，心跳超时阈值不能太紧，否则误判重连频繁。

### 13.4 幂等窗口不是永久一致性

Redis TTL 幂等可以解决短期重发重复，但不是永久幂等审计。若后续要求跨天重试一致性，再评估 DB 方案。

### 13.5 sync 乱序与大批量补偿

若长时间离线后增量过大，需要支持分批返回 `hasMore`，否则单次响应过重。

### 13.6 token 续期与实时连接耦合

若 WebSocket 使用过期 token，重连会反复失败。需要与现有 refresh token 逻辑协同，避免死循环。

## 14. 测试方案

遵循仓库规则，本方案不允许只改实现不补测试。

## 14.1 第一阶段测试

### 前端单测

* 单例 manager 不重复建连
* 心跳发送与 pong 超时判定
* `offline/online`
* `visibilitychange`
* 指数退避重连次数与上限

### 后端单测

* 收到 `ping` 返回 `pong`
* session 建立、替换、移除
* 鉴权失败返回统一 `error`

## 14.2 第二阶段测试

### 前端单测

* envelope 编解码
* 非法 type / 非法 data 忽略或报错

### 后端单测

* envelope 正确分发到对应 handler
* 旧 `CHAT_*` type 兼容映射

## 14.3 第三阶段测试

### 前端单测

* 发送消息初始为 `sending`
* 收到 `message_ack` 后变 `sent`
* ACK 超时后变 `failed`
* 重试后不新增重复消息

### 后端单测

* 同一 `clientMessageId` 重复发送只落库一次
* ACK 回包包含 `clientMessageId`

### 集成测试

* 文本消息发送 -> ACK -> 对端接收
* ACK 丢失 -> 重连 -> sync 后本地状态自愈

## 14.4 第四阶段测试

### 前端集成测试

* 重连成功后自动发 `sync_request`
* `sync_response` 正确合并历史缺口
* 未知会话消息进入后列表可见

### 后端集成测试

* `sinceCursor` 增量查询正确
* `hasMore / nextCursor` 正确

### 浏览器 E2E

必须至少覆盖：

1. 登录后建立实时连接
2. 断网后进入 offline
3. 恢复网络后自动重连
4. 发送中的消息收到 ACK 后转 sent
5. 对端在线消息实时到达
6. 重连后漏掉的消息通过 sync 补齐

## 15. 推荐实施节奏

建议拆成四个独立 PR：

1. `feat(realtime): add singleton websocket manager and connection state machine`
2. `refactor(realtime): unify websocket envelope protocol`
3. `feat(message): add client message id and ack dedup flow`
4. `feat(sync): support reconnect incremental message sync`

这样做的好处是：

* 每个 PR 风险可控
* 每阶段都能独立回滚
* 不会把连接治理、协议治理、消息一致性、同步补偿混成一个大改

---

## 16. 结论

这套演进方案的核心不是一次性推翻现有实现，而是：

1. 先把连接层做稳
2. 再把协议层做统一
3. 再把消息发送态与 ACK 做清楚
4. 最后补上重连后的增量同步

这样可以在 **不破坏当前聊天主链路、不直接改 schema、不绕过现有 Service 分层** 的前提下，把 HideChat 从“能用的实时聊天”逐步升级为“更稳定、可维护、可自愈的类微信实时聊天架构”。
